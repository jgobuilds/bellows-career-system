# Code Review — Bellows (2026-07-19)

**Reviewer:** engineering-standard · **Scope reviewed:** the tracked repo — `engine/` (27 Python files, 3,403 LOC), `config.py`, `setup.py`, `tests/` (26 tests), and `engine/hub.html` (the single-page view). `personal/` is gitignored and was excluded. · **Branch:** `main` (post-refactor: jobkey, pipeline_store, docx_common, _paths, honest docx_finalize).

## 1. Executive summary

Bellows is a **well-architected small system** — clean layering (config → engine tools → repository → server → view), real design patterns (Repository, Strategy/registry, Factory), and exceptional module docstrings. The recent refactor pass paid down most of the structural debt. The single biggest risk is not the code, it's that **nothing runs automatically**: 26 good unit tests and three linters exist, but no CI gates anything, so quality depends on remembering to run it — and the linters just caught a latent bug (`set_notes` writes an unsanitized note) that unit tests missed. The biggest strength is architecture + documentation. **Headline recommendation: add a CI workflow that runs the tests + `ruff` on every push, then fix the `set_notes` bug.** That one automation move lifts the whole repo from "Defined" to "Managed."

**Overall maturity: 3/5.** Consistent, well-reasoned approaches exist and are mostly followed, but they aren't yet enforced by automation — the L3→L4 jump is a single CI workflow.

## 3. What's working (keep this)

- **The layering is real and the dependency direction is clean.** `jobkey` is pure (no imports), `pipeline_store` → `config`, `server` → `pipeline_store`, view is separate. Business logic is *not* smeared into request handlers.
- **Genuine patterns, not cargo-culted:** the `pipeline_store` Repository, the `ats_sweep.FETCHERS` Strategy/registry (a new board = one function + one dict entry), and the config-driven `terms_to_regex` Factory. One source of truth for job identity (`jobkey`).
- **Documentation is a strength** — every module opens with a "what it is / what it is NOT" docstring, plus README/SETUP/starter/tests/architecture-review guides. This is well above average.
- **The tests that exist are good** — deterministic, fast, assert real behavior on the core logic, and pin the exact bugs that motivated the refactors.

## 4. Findings by severity

### 🔴 Critical
None. No data-loss, security, or ships-broken correctness defects.

### 🟠 High
- **[H1] No CI/automation — quality depends on memory.** The 26 unit tests and `ruff`/`mypy`/`bandit` gate nothing; there is no `.github/workflows`. *Why it matters:* regressions land silently — the `set_notes` bug (M1) is live proof of what slips through when nothing runs on push. *Fix:* add a GitHub Actions `quality.yml` running `python -m unittest discover -s tests` + `ruff check`. This is Mode B and unblocks everything else.
- **[H2] The `_paths` sys.path injection defeats static analysis.** `engine/_paths.py` mutates `sys.path` at runtime so `import config` resolves; mypy can't follow it, so **33 of 43 mypy findings are `import-not-found`** and type-checking is effectively off (and it's the source of the 14 `E402` warnings). *Why it matters:* you can't turn on a type gate while imports don't statically resolve. *Fix:* make `engine/` a real installable package — a minimal `pyproject.toml` + `pip install -e .` — and drop the shim; imports resolve statically and mypy comes alive. (This is the mature form of F3: that refactor removed the *copy-paste* bootstraps but kept a tooling-hostile runtime trick.)

### 🟡 Medium
- **[M1] `set_notes` writes an unsanitized note (latent bug).** `server.py:350` computes `safe` (escapes `\`, `"`, newline→space, `|`→`/`) but `server.py:368` writes the **raw** `notes` into pipeline.md: `f"- **Notes:** {notes}\n"`. *Why it matters:* a multi-line note breaks the `### Job N` detail block (splits across lines). Caught by `ruff F841` (`safe` unused). *Fix:* use `safe` in the newline. (jobs.json uses raw `notes`, which is fine — JSON escapes.)
- **[M2] Testing isn't automated and has gaps.** Test:prod LOC ratio 0.09; only the pure core is covered — the server endpoints and the datastore round-trip (`set_status` → jobs.json + pipeline.md) have no tests. *Fix:* run tests in CI (H1) + add ~2 integration tests (a `set_status` round-trip, a `jobs_payload` docs check) + a coverage report.
- **[M3] Observability is `print()`-only.** No `logging` module, levels, or structured output anywhere (`server`, both sweeps). *Why it matters:* no quiet/verbose control, no log file, hard to diagnose a bad sweep after the fact. *Fix:* adopt `logging` with levels — it maps cleanly onto the sweep's existing `verbose` flag; low effort.
- **[M4] Loose dependency pins, no lockfile.** `requirements.txt` uses `>=` and there's no lock. *Fix:* pin exact versions or add a lock (`uv`/`pip-tools`) so a fresh install is reproducible.

### 🔵 Low
- **[L1] Three remaining silent `except`s** (`bandit B110`): `ats_match.py:197` (the config-override), `ats_sweep.py:188` (`_enrich_desc`), `server.py:569`. Narrow to the expected type or log — don't swallow silently.
- **[L2] The local server serves the entire repo root.** `main()` does `os.chdir(ROOT)`, so the whole tree — including `personal/` — is reachable over `127.0.0.1`. It's localhost-only and single-user, but it's broader than needed. *Fix (defense in depth):* serve only `engine/` + `personal/applications/` doc paths.
- **[L3] Dead variable `end` in `set_notes`** (`server.py:362`, `ruff F841`) — computed, never used. Remove.

### ⚪ Nits (auto-fixable / taste)
- **[N1]** `E741` ambiguous `l` (12×, e.g. `pipeline_store.py:79`, `add_job.py:72`) — rename `l` → `line`.
- **[N2]** `E731` lambda-assignment (4×: `docx_finalize.py:44`, `server.py:257`, `ats_sweep.py:381/390`) — convert to `def` (or leave; they're tiny).
- **[N3]** `E702` semicolon-joined statements (13×) — split; `ruff check --fix` handles most.
- **[N4]** `E402` imports-not-at-top (14×) — all the `import _paths` pattern; resolved wholesale by H2, or `# noqa: E402` until then.

## 5. Cross-cutting themes

1. **Everything is manual.** The tests, the linters, the formatter — all one command away, none automated. The entire repo is one CI workflow short of "Managed." This is the theme behind H1, M2, and the `set_notes` bug all at once.
2. **A runtime trick vs. static tooling.** The `_paths`/`sys.path` pattern (H2) is DRY at runtime but blinds mypy and floods `E402`. The packaged form is strictly better and unlocks type-checking.
3. **`print` as observability (M3)** — consistent, but floor-level; a local tool still benefits from `logging` levels.
